## Roadmap

このロードマップは、Unity向け「モジュール型ゲーム基盤」としての整合性を高めつつ、段階的に機能を厚くしていくための指針です。時期は目安であり、優先度は「開発体験（DX）」「品質（テスト容易性）」「拡張性（Provider差し替え）」を重視して調整します。

---

## 0. 現状（フェーズの基礎実装は完了）

現段階は、下記フェーズ1〜4の **基本となる実装（骨格・最小動作・テストの入口）を完了している状態** です。以降は、各フェーズ内の未実装領域を「実運用レベル」に引き上げ、長期的な拡張計画を並行して進めます。

---

## 1. フェーズ構成（モジュール単位）

### フェーズ1（Core / Resource / Pool / Debug）

- **狙い**: 全モジュールの土台となる「初期化・参照管理・再利用・診断」を安定させる
- **未実装 / 深化（短〜中期）**
  - Project Bootstrap の一本化（初期化順序の固定、未初期化利用の防止）
  - 非同期初期化の失敗戦略（例外/タイムアウト/フォールバック、ログ方針）
  - Resource（Addressablesラップ）のハンドル管理と解放規律の明確化
  - Pool の計測（ヒット率、最大数、リーク検出）とチューニング導線
  - Debug のカテゴリ/レベル/出力先（Console/File）の統一とフィルタリング

### フェーズ2（UI / Localization / Time / Scene）

- **狙い**: 画面遷移・時間・シーン・言語切替を「運用しやすい規約」で揃える
- **未実装 / 深化（短〜中期）**
  - UIの運用モデル確立（Page遷移、Dialogスタック/キュー、戻る制御、連打対策）
  - UIアニメーションの統一インターフェースとプリセット運用
  - Localization のロード形式（CSV/JSON）とキャッシュ、言語切替の通知設計
  - Time（タイマー/時間スケール）のゲーム進行への適用指針
  - Scene 管理（ロード/アンロード、フェード、アドレッサブル連携）の統一

### フェーズ3（MasterData / SaveData / Audio / Network）

- **狙い**: データ運用（読み込み/保存/通信/音）を「壊れにくい」形にする
- **未実装 / 深化（短〜中期）**
  - MasterData のロード・キャッシュ・検索APIの強化（生成コードの安定化含む）
  - SaveData の実運用ルート整備（ストレージ/シリアライズ/暗号化の境界、復旧戦略）
  - Audio の実戦機能（ボリューム永続化、Ducking、同時発音、Mixer運用）
  - Network の共通化（Request/Response基底、エラー表現、リトライ/タイムアウト/キャンセル）
  - 環境切替（dev/stg/prod）と通信ログ（相関ID等）の整備

### フェーズ4（FSM / SDK）

- **狙い**: ゲーム固有の進行・外部SDK連携を「差し替え可能な抽象」で支える
- **未実装 / 深化（中期）**
  - FSM の拡張（状態/遷移の拡張ポイント、デバッグ可視化、運用ガイド）
  - SDK Provider 実装の追加（Ads/Analytics/Store の具体実装例）
  - オフライン/失敗時設計（SDK初期化失敗時の影響制御、復帰戦略）

## 2. 横断テーマ（全フェーズ共通）

### 2.1 Settings（運用の背骨）

- **目的**: Settingsアセットが増えても運用が破綻しない状態を作る
- **計画**
  - Settingsの検証（必須項目未設定の検出）
  - CI向け検証コマンド（Editorバッチでの事前チェック）
  - Settingsのスキーマ互換（バージョン管理、移行手順）

---

## 3. 開発体験（DX）と品質（優先度 中〜高）

### 3.1 テストの層分けと自動化

- **EditMode**: Unity依存の薄い層（変換/暗号化/ルール）を中心に増強
- **PlayMode**: UI・シーン・Audio・Network（モック）をシナリオで検証
- **CI**: バッチ実行（最低限の回帰を常時担保）

### 3.2 ログ / 診断の体系化

- ログカテゴリ（Network/SaveData/UI等）・レベル・出力先（Console/File）
- 障害解析しやすいイベントID/相関ID（通信トレース）

---

## 4. 長期（優先度 低、価値は高い）

- **アーキテクチャのテンプレート化**
  - サンプルゲーム（小規模）で全モジュールの接続例を提供
- **パフォーマンス最適化**
  - Addressablesロードとメモリ解放の規律、GC圧の低減
- **拡張UI**
  - リスト仮想化の強化、UIの入力/フォーカス管理の統一

---

## 互換性・破壊的変更ポリシー

- 当面は **minor でも破壊的変更が起こり得る** 前提で開発します。
- 破壊的変更は `ROADMAP.md` と変更履歴に記録し、移行手順を明示する方針です。
